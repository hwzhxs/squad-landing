Mobile Agentic Browser

Mobile browser 15 年没变过。50 亿设备上运行着为「看网页」设计的工具，但用户真正想做的是「完成任务」。Agentic AI 正在改写这个等式。

Chapter 01

用户在 Mobile Browser 里到底在做什么

在谈设计之前，必须先回答一个基本问题：人们打开手机浏览器的时候，到底在做什么？已有的用户研究给出了清晰的画像。

核心行为模式

搜索 → 行动（Search-to-Action）是 mobile browser 的第一用途。68% 的用户在手机上开始购物调研，即使最终在桌面完成购买（Google Multichannel Study）。82% 的人在重大购买前会在手机上搜索。55-60% 的 Google 搜索发生在 mobile 上。

本地发现（Local Discovery）是第二大场景。88% 的 "near me" 搜索发生在手机上，76% 在当天产生线下到访。这是 desktop 几乎不存在的场景——人在路上、在商场、在餐厅门口，需要即时答案。

社交驱动的浏览（Social-Driven Browsing）是第三大来源。31% 的 mobile web session 发生在 App 内置浏览器（Instagram、微信、微博的内嵌 webview）。用户从社交内容点击链接 → 进入浏览器 → 完成购买或获取信息。

碎片化消费（Snacking）是第四种模式。Mobile 用户平均每次访问 3 个页面，停留时间 700-775 秒，远低于 desktop 的 7 个页面和 996-1918 秒。Mobile browsing 是快速、浅层、频繁的。

关键洞察：Mobile browser 的用户不是在"浏览"——他们在寻找答案、做决策、完成交易。但浏览器给他们的工具仍然是「打开一个又一个网页」。92% 的时间用户选择留在 App 里，因为 App 能直接帮他们做事。Browser 只剩下 App 做不了的事：开放 web 上的搜索和比较。

五个核心痛点

  1. 01
    信息过载，找不到答案。用户想要答案，得到 10 个蓝色链接。Mobile 上更痛——屏幕小，每个结果要点进去、等加载、扫读、返回、再试。一个简单的比价任务，desktop 3 分钟，mobile 10 分钟。
  2. 02
    多步任务断裂。订机票 = 搜航班 → 比价 → 选座 → 填信息 → 支付。每步都是独立页面，中间被广告、弹窗、登录墙打断。Mobile cart abandonment rate 高达 70%（Baymard 2025），比 desktop 高 24 个百分点。
  3. 03
    Tab 管理是灾难。平均用户打开 12-15 个 tab，90% 再也不会回去看。Mobile 上 tab 切换体验极差。但没人敢关，因为"可能还要用"。
  4. 04
    上下文不断丢失。研究一个话题，打开 8 个页面，读了半小时。第二天想继续——全忘了。浏览器只记 URL，不记你学到了什么。
  5. 05
    隐私焦虑 vs 便利的永恒矛盾。每个页面平均 17 个 tracker，cookie banner 占半个屏幕。想要便利就要出卖隐私，想要隐私就要忍受不便。
Chapter 02

市场格局：巨头入场，窗口正在关闭

2025 是 agentic browser 元年。四个重量级玩家同时入场，但全部 desktop first——mobile agentic browser 目前是一片蓝海。

关键市场信号

  1. 01
    Dia 被 Atlassian $6.1 亿收购（2025.10）——独立消费级 AI browser 的商业可行性受到质疑。Arc 的教训：创新 UI 很酷但大众不买单。Dia 从 C 端转向企业 B2B。
  2. 02
    全部 desktop first——Atlas macOS only，Comet Windows + Mac，Neon desktop only。Mobile 版全在 "coming soon"。没有人在 mobile 上认真做 agentic browser。
  3. 03
    安全事件频发——Comet 的 CometJacking 漏洞（prompt injection）、MCP API 默认开放本地命令执行。Agentic = 更大的攻击面。
  4. 04
    Chrome 71% + Safari 20% = 91%——mobile browser 市场高度集中。Google 和 Apple 一定会在 Chrome/Safari 里加 agent 能力。但大象转身慢，且受广告模式和平台政策制约。
Chapter 03

关键问题：为什么是 Browser，不是 ChatGPT App？

这是最重要的战略问题。如果 ChatGPT App 已经能上网、能搜索、能执行任务——为什么用户还需要一个 agentic browser？如果回答不了这个问题，产品就没有存在的理由。

核心论点：AI App（ChatGPT/Claude）和 Agentic Browser 的边界，在于「谁拥有上下文」。AI App 从零开始，每次都要你描述问题。Browser 已经在你的问题现场。

四个本质区别

1. 上下文拥有权不同

ChatGPT App：你需要把问题描述给 AI。"帮我在京东上找一个 500 块以下的空气炸锅"——AI 要自己打开京东、搜索、理解页面。

Agentic Browser：你已经在京东的搜索结果页上了。AI 直接看到你看到的东西。"帮我比较这几个" 就够了。Browser 拥有实时的、第一人称的 web 上下文。ChatGPT Agent 拥有的是一个隔离的虚拟机里的模拟浏览器。

2. 操作对象不同

ChatGPT Agent 操作的是一个沙盒环境——它在自己的虚拟电脑上打开网页、截图、点击。用户的 cookie、登录状态、历史记录，它都没有。所以它无法帮你在已登录的淘宝里下单，无法看到你的 Gmail 里的邮件。

Agentic Browser 操作的是用户真实的浏览环境——你的登录状态、你的购物车、你的收藏夹。它能在你已经登录的网站上直接行动。这是 Consumer Reports 的评测（2025）明确指出的关键区别。

3. 交互模型不同

AI App 是对话驱动的——你说，它做，你看结果，你再说。这是一个串行的、语言中心的交互。

Browser 是混合驱动的——有时候你自己浏览，有时候让 AI 接管，有时候你和 AI 同时在一个页面上工作。Atlas 的设计就是这样：用户可以随时从 Agent Mode 切回手动浏览，无缝衔接。

这个区别在 mobile 上更重要。在手机上打开 ChatGPT 描述一个购物需求，不如直接在购物页面上说 "帮我比较"。

4. 入口位置不同

AI App 是一个独立入口——你要专门打开它。Browser 是你已经在用的东西。Data.ai 数据显示 mobile 上 92% 的时间在 App 里，但 64% 的 web traffic 来自 mobile——说明人们频繁地在 App 和 Browser 之间切换。

Browser 是所有 web 行为的必经之路，AI App 是你主动去找的额外工具。Agent 能力放在 browser 里 = 零额外步骤。

所以边界是什么？

维度 AI App（ChatGPT） Agentic Browser
最佳场景 从零开始的研究、写作、分析 基于当前浏览的即时任务
上下文 用户描述 / connector 授权 实时页面 + 登录态 + 历史
操作能力 沙盒虚拟机，无用户态 真实浏览器，有用户态
交互 对话串行 手动 + AI 混合，随时切换
安全风险 低（隔离沙盒） 高（真实环境，可操作敏感数据）
Mobile 优势 独立入口，需主动打开 已在用，零切换成本
结论：AI App 适合"我想做一件新的事"。Agentic Browser 适合"我正在做一件事，帮我做得更好更快"。两者不是替代关系，是互补关系。但在 mobile 上，browser 的优势更大——因为 mobile 用户的核心行为就是在浏览中触发行动。
Chapter 04

五个设计原则

Principle 01

Intent-First, Not URL-First

地址栏是 1994 年的设计。输入界面应该是意图捕获器：支持自然语言、语音、截图。用户不想输入 URL，他们想说 "帮我找今晚苏州最好的日料，人均 300 以内"。

Principle 02

Task State > Page State

核心 UI 从「你打开了哪些页面」变成「你有哪些进行中的任务」。Tab bar → Task bar。不是 "淘宝 - 京东"，而是 "比价空气炸锅（进行中）- 订餐厅（等你确认）"。

Principle 03

Progressive Autonomy

三级控制：L1 Suggest（AI 建议，你执行）→ L2 Execute（AI 执行，你确认关键节点）→ L3 Autopilot（全自动，高风险时暂停）。Per-task 可调，随信任升级。

Principle 04

Transparent Agency

两层透明：What（agent 当前在做什么）+ Why（为什么做这个决策）。默认只显示 What，点击展开 Why。让用户感到"在掌控"，而非"被代替"。

Principle 05

Memory as a Feature

浏览器 history 只是 URL 列表。Agentic browser 应记住语义：你上周研究了什么、对比了哪些选项、在哪个页面读到了关键信息。"继续我上周的调研" 应该是一个按钮。浏览不再一次性，而是可积累的。

Chapter 05

三个战略方向：Mobile 特有的机会

Desktop agentic browser 的战场已拥挤。但 Mobile 有三个 Desktop 没有的独特机会。

方向 A：碎片时间的任务完成者

Mobile 用户的使用是碎片化的——地铁上 3 分钟、等咖啡 2 分钟。这些碎片时间里，传统浏览器完不成任何多步任务。

机会：用 agent 把 10 分钟的任务压缩到 1 分钟用户时间。30 秒启动 + 30 秒确认结果，中间 9 分钟让 agent 后台完成。

方向 B：语音驱动的 Hands-Free Browsing

Mobile 有麦克风和耳机。开车、做饭、健身时手不空但脑子空着。全球超过 1/5 的人口已经在使用语音搜索（DataReportal 2024）。

机会：纯语音 browsing。"帮我看看明天苏州天气" → agent 搜索 → 语音播报。不需要看屏幕。Arc Search 已有 "举起手机到耳边 = 开始语音对话" 的交互。

方向 C：跨信息源的统一任务层

这是最大的机会也是最难的。Mobile 上完成一个任务要在多个 app/网站之间跳转（浏览器搜餐厅 → 大众点评看评价 → 微信问朋友 → 高德导航）。每次跳转都丢失上下文。

机会：Browser 变成任务的"中控台"，agent 协调多个信息源，用户只在 browser 里看到统一的任务视图。不替代其他 app，而是整合它们的信息。

Chapter 06

具体 Concept：Task Flow Browser

核心隐喻：浏览器不是「窗口」，是「助理的工作台」。

界面结构

底部
意图输入区

输入框 + 麦克风 + 相机。自然语言 / URL / 语音 / 截图识别。AI 理解意图后决定行动。

中间
任务卡片流

每个任务 = 一张卡片。⏳执行中 / 🔔需确认 / ✅已完成。左滑归档 / 右滑追加指令。

顶部
智能上下文栏

当前任务名 + agent 状态 + 可信度指示器。展开看引用来源和评分。

四个关键交互

  1. 01
    任务启动 — 用户说 "帮我比较 iPhone 16 和 Pixel 9 的相机" → AI 创建任务卡片 → 显示进度 "正在收集评测数据（3/7 来源）..." → 完成后：结构化对比表 + 结论 + 来源链接。
  2. 02
    后台执行 + 推送回调 — 用户启动任务后退出 app → Agent 继续后台执行 → 完成后推送 "你的机票比价完成了，找到 4 个选项" → 点通知直接看结果卡片。
  3. 03
    确认门控 — Agent 准备下单 → 卡片变 🔔 "准备在京东下单，¥299，确认？" → 一键确认或修改。高风险操作永远不自动执行。
  4. 04
    记忆续接 — 第二天打开 app → "你昨天在研究空气炸锅，要继续吗？" → 恢复完整上下文：之前对比的商品、你的倾向、待决定的问题。

Chapter 07

风险、诚实的不确定性，与结论

四个必须面对的风险

  1. 01
    iOS 后台限制 — Background App Refresh 不可靠，agent 长任务需要服务端执行 + 设备同步。这增加了架构复杂度和成本。
  2. 02
    中国市场的封闭生态 — 微信/淘宝/大众点评都是围墙花园，跨 app 信息整合在中国比欧美更难。
  3. 03
    商业可行性 — Dia 被收购的教训：独立消费级 AI browser 生存困难。需要大模型或搜索引擎的资源后盾，或找到差异化的盈利模式。
  4. 04
    安全 — Agent 权限 = 更大的攻击面。Comet 的 CometJacking、MCP API 漏洞是前车之鉴。安全设计必须从 Day 1 严格，不能后补。
最终判断：

Mobile browser 15 年没本质变化。现在变的时候到了——不是因为技术成熟了，而是因为用户的痛点大到无法忽视（70% 的购物车在手机上被放弃），同时 AI 能力刚好跨过了"能替你操作网页"的门槛。

窗口期约 12-18 个月。Google 和 Apple 一定会把 agent 塞进 Chrome 和 Safari。但它们的 agent 受限于广告模型和平台政策。这给独立产品留了一条缝隙——做一个真正站在用户这边的 mobile agentic browser。

关键不是"做不做"，而是"有没有差异化的资源优势"来支撑这个产品活过 Google/Apple 的追赶期。
